# Lexis: Programmatic Verification of the Issue Type

> **Prefix:** `lex-` | **Type:** Unbreakable Law | **Scope:** Every Issue created in Guardia repositories that participates in the Issue-Driven flow

## Purpose

`lex-issue-quality` requires every Issue to have a defined Issue Type (`Feature`, `Task`, `Bug`, `Epic`). The native GitHub Issue Type field is populated automatically when an Issue is created via template (`.github/ISSUE_TEMPLATE/*.yml` declares `type:`), but it is **NOT** populated when an Issue is created via `gh issue create` without a template. Without programmatic post-creation verification, Issues created via CLI remain typeless â€” which breaks the HARD-GATE in `lex-agent-planning` and disrupts the owners-by-type table. This Law codifies the mandatory verification.

## Law

> **Every agent that creates an Issue (human or AI, via UI/CLI/MCP) MUST programmatically verify, immediately after creation, that the Issue's native `type` field is populated with a value compatible with the template used (`Feature` for `feature-request` / `user-story-for-api` / `user-story-for-frontend`; `Task` for `tech-task` / `plan`; `Bug` for `bug`; `Epic` for `epic`). If empty, the agent MUST apply the type via `gh api -X PATCH repos/{owner}/{repo}/issues/{N} -f type={Feature|Task|Bug|Epic}` before any subsequent transition on the Issue. Applying the `status: todo` label (per `lex-issue-status` Axis A) on an Issue without a populated `type` is FORBIDDEN.**

## Scope

- **Applies to:** every Issue created in Guardia repositories, regardless of mechanism (GitHub UI, `gh issue create`, MCP `create_issue`, automated scripts).
- **Bound agents:** `warrior-eunomia` (top-level and Plan sub-issue modes), `warrior-athena` (when delegating child Issue creation), `warrior-calliope` (Epic decomposition), and any agent that invokes `kata-plan-task`, `kata-decompose-issue-into-plans`, or `kata-contributing-issue`.
- **Exceptions:** Issues generated by Dependabot or security scanners follow their own flow and are exempt.

## Rules

### 1. Mandatory post-creation verification

Immediately after creating the Issue, the agent **MUST**:

```bash
gh api repos/{owner}/{repo}/issues/{N} --jq '.type.name // empty'
```

- If it returns a value (`Feature`, `Task`, `Bug`, `Epic`) â†’ proceed to compatibility check (Rule 2).
- If it returns empty â†’ apply manually (Rule 3).

### 2. Template â†” type compatibility

| Template | Accepted Issue Type |
|---|---|
| `feature-request` | `Feature` |
| `epic` | `Epic` |
| `user-story-for-api` | `Feature` |
| `user-story-for-frontend` | `Feature` |
| `bug` | `Bug` |
| `tech-task` | `Task` |
| `plan` | `Task` |

If the returned type is incompatible with the template, **abort and alert the user** â€” do not silently rewrite (it may mask a creation error).

### 3. Manual application when absent

If the verification returns empty:

```bash
gh api -X PATCH repos/{owner}/{repo}/issues/{N} -f type={Feature|Task|Bug|Epic}
```

Then re-verify (Rule 1) to confirm persistence.

### 4. Precondition for transitions

Applying the `status: todo` label (entry into Axis A of `lex-issue-status`) **ALWAYS** happens **AFTER** the type is populated and verified. The HARD-GATE precondition (b) in `lex-agent-planning` requires this explicitly.

## HARD-GATE

Per [`lex-hard-gate-pattern`](../quality/lex-hard-gate-pattern.md):

```
<HARD-GATE>
warrior-eunomia, warrior-athena, warrior-calliope and any agent that
creates an Issue MUST NOT apply the `status: todo` label (entry into
Axis A of lex-issue-status) without satisfying ALL criteria:

  (a) `gh api repos/{owner}/{repo}/issues/{N} --jq '.type.name'` returns
      a non-empty value
  (b) Returned value is one of: Feature | Task | Bug | Epic
  (c) Value is compatible with the template used per Rule 2 of this Lex

This rule applies to EVERY Issue in the Issue-Driven flow, regardless of:
  - perceived size ("it is just a chore")
  - urgency ("production fire")
  - who requested ("the CEO asked")
  - team confidence ("we already tested a lot")

Declared exception: Issues generated by Dependabot or security scanners
follow their own flow. Every other Issue in the flow respects the gate.
</HARD-GATE>
```
