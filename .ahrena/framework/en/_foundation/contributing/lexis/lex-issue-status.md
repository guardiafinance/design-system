# Lexis: Canonical Status Labels on Issue and PR

> **Prefix:** `lex-` | **Type:** Unbreakable Law | **Scope:** Issues and Pull Requests in Guardia repositories that participate in the Issue-Driven flow

## Purpose

The Issue body (canonical), the Issue, and the PR carry the same work at distinct moments of the cycle. Without a canonical set of status labels, the agent loses the cross-reference, dashboards drift, and the aggregated parent↔Plan sub-issue computation becomes imprecise. This Law codifies the `status: <name>` labels that mirror the enum in `lex-agent-planning`, **splits them into two disjoint axes** (dev cycle and release cycle), guarantees intra-artifact consistency, and keeps orthogonality with the Discovery labels (`pending-spec`/`spec-ready`).

## Law

> **Every Issue and every PR that participates in the Issue-Driven flow MUST carry exactly one `status: <name>` label from the canonical set. The set splits into two disjoint axes: **Axis A (dev cycle)** applicable to feature/fix/chore/refactor Issues/PRs (`status: todo`, `status: development`, `status: to review`, `status: review`, `status: done`); **Axis B (release cycle)** applicable exclusively to the dedicated release Issue created by Janus (`status: to release`, `status: release`, `status: done`). The terminal `status: abandoned` is shared by both axes. Mutex is **intra-artifact**: no Issue or PR carries two `status:*` labels simultaneously. Applying an Axis B label to an Axis A Issue/PR (or vice versa) is FORBIDDEN. The `status:*` label is orthogonal to the Discovery labels (`pending-spec`/`spec-ready`) and to the type labels (`feature request ➕`, etc.) — multiple axes coexist.**

## Coverage

- **Applies to:** every Issue opened via approved templates (`feature-request`, `user-story-for-api`, `user-story-for-frontend`, `tech-task`, `bug`, `plan`, `release`), and every PR in Guardia repositories. Plan sub-issues (created via the `plan` template per `lex-agent-planning`) follow Axis A like any other dev-cycle artifact.
- **Bound agents:**
  - Axis A — `warrior-eunomia` (`— → todo`), `warrior-athena` (`todo → development`, `development → to review`, `to review → done`), `warrior-argos` (`to review ↔ review`).
  - Axis B — `warrior-janus` (`— → to release`, `to release → release`, `release → done`).
- **Exceptions:**
  - **Epic** does not receive `status:*` — it is decomposed into child Issues, each with its own cycle (Axis A). The Epic closes when all children (`Tracked by`) reach `done`.
  - **Issues generated by Dependabot** or security scanners follow their own flow and are exempt.

## Rules

### 1. Axis A — Dev cycle (feature/fix/chore/refactor Issues/PRs)

| Label | Suggested color | When to apply | Owner |
|---|---|---|---|
| `status: todo` | `#cccccc` (light gray) | Canonical plan in the Issue body, branch linked, worktree ready | `warrior-eunomia` (fallback: session agent) |
| `status: development` | `#83d2ff` (light blue) | Implementation in progress (Athena Phase 4) | `warrior-athena` |
| `status: to review` | `#fff3a3` (light yellow) | PR opened, waiting for a reviewer to pick it up | `warrior-athena` (entry); `warrior-argos` (return from `review`) |
| `status: review` | `#fbca04` (yellow) | Argos or human actively reviewing | `warrior-argos` |
| `status: done` | `#0e8a16` (green) | PR merged; Issue closed via `Closes #N` | `warrior-athena` (on merge) |
| `status: abandoned` | `#6e6e6e` (dark gray) | Plan discarded (alternative terminal) | creator or current owner |

### 2. Axis B — Release cycle (exclusively the dedicated release Issue)

| Label | Suggested color | When to apply | Owner |
|---|---|---|---|
| `status: to release` | `#ffb178` (light orange) | Janus opened release Issue; populated `Tracks: #N1, #N2, ...` | `warrior-janus` |
| `status: release` | `#e07400` (orange) | `kata-release-prepare` running; human approved bump/changelog | `warrior-janus` |
| `status: done` | `#0e8a16` (green) | Tag pushed, `validate-tag.yml` passed, Release published | `warrior-janus` |
| `status: abandoned` | `#6e6e6e` (dark gray) | Release aborted before tag | `warrior-janus` |

The release Issue is created by Janus as the entry point of the release cycle. There is no "release branch": the cycle operates on the release Issue + tag + GitHub Release. Details in `warrior-janus` and `kata-release-prepare`.

### 3. Intra-artifact mutex

At any moment, **the same Issue (or PR)** carries exactly one `status:*` label. Applying two `status:*` labels simultaneously to the same Issue/PR (e.g., `status: to review` + `status: review`) is FORBIDDEN.

Mutex is **not** cross-artifact: the release Issue (Axis B) coexists with N feature Issues in `done` (Axis A) without conflict — they are distinct artifacts.

Each transition is executed by:

```bash
gh issue edit {N} --remove-label "status: <previous>" --add-label "status: <next>"
gh pr edit {N}    --remove-label "status: <previous>" --add-label "status: <next>"
```

Prefer MCP `update_issue` / `update_pull_request` when the GitHub MCP server is listed in `mcp.servers` and active (per `lex-mcp` rule 1).

### 4. Cross-cycle labeling forbidden

Axis B labels (`status: to release`, `status: release`) **may not** be applied to feature/fix/chore/refactor Issues/PRs. Axis A labels (`status: todo`, `status: development`, `status: to review`, `status: review`) **may not** be applied to a release Issue.

Detection: the Issue type (`gh issue view {N} --json type`) determines the permitted axis. The release Issue carries Issue Type `Task` (or `Feature` when created as a release feature by template convention) + label `release ↗️`. Cross-cycle labeling is a HARD-GATE violation.

### 5. Issue ↔ PR synchronization

When a PR is opened for an Issue in `status: development`, the agent executing the transition (per `lex-agent-planning` Table A) MUST simultaneously update:

1. The `status: <name>` label on the Issue.
2. The `status: <name>` label on the PR.
3. Trigger `kata-flush-plan-to-issue` to ensure the Issue body reflects the post-transition state.

Failure in any of the three produces drift detectable at Gate 2 (per `kata-quality-gate`).

### 6. Orthogonality with Discovery labels

Discovery labels (`pending-spec`, `spec-ready`, defined by plan-038) operate on a separate axis:

- `pending-spec`/`spec-ready` control **entry** into the Athena flow for the US-child (User Story child of an Epic). They coexist with `status:*` on the same Issue.
- A US-child created by Calliope is born with `pending-spec` and **without** `status:*`. It receives `status: todo` only when it earns `spec-ready` (transition done by the corresponding PM after producing the spec).
- Bug and Tech-task skip the spec gate and receive `status: todo` directly on creation by Eunomia.

### 7. Epic does not receive `status:*`

An Epic is decomposed by Calliope (plan-038) and never passes through Athena directly. The Epic has no `todo → development → ...` cycle of its own; its state is derived from `Tracked by` (children with `status:*`). Applying `status:*` to an Epic is FORBIDDEN.

### 8. Initial label creation in the repository

Each repository that adopts the flow MUST create the labels via `gh label create` (idempotent script in `scripts/bootstrap_labels.sh`). The script bootstraps the full label catalog (type + workflow + size + cross-cutting) idempotently.

### 9. Absence of `status:*` on a non-Epic Issue in the flow is a violation

Every non-Epic Issue that participates in the Issue-Driven flow MUST carry exactly one `status:*` label from the canonical set of the correct axis. **Absence** of any `status:*` label on a non-Epic Issue active in the flow is FORBIDDEN — this closes the silent gap between "created" and "in the flow".

Detected by:

- `kata-contributing-issue` applies `status: todo` as the final step of creation (per `lex-issue-quality` Rule 6).
- Templates `.github/ISSUE_TEMPLATE/*.yml` declare `labels: [..., "status: todo"]` to auto-apply on creation via UI.
- Periodic verification detects open Issues without `status:*` (excluding Epic) and signals the owner to regularize.

Issues without `status:*` do not satisfy the `lex-issue-quality` creation HARD-GATE precondition (d) and block branch/PR.

## HARD-GATE

Per [`lex-hard-gate-pattern`](../quality/lex-hard-gate-pattern.md), the textual block of this Lex is canonically expressed as:

```
<HARD-GATE>
warrior-eunomia, warrior-athena, warrior-argos, warrior-janus, and any
other agent MUST NOT apply a `status:*` label to an Issue or PR without
satisfying ALL criteria:

  (a) Label belongs to the canonical set of the correct axis:
      - Axis A (feature/fix/chore Issues/PRs): todo | development |
        to review | review | done | abandoned
      - Axis B (release Issue exclusively): to release | release |
        done | abandoned
  (b) Issue/PR is not in a terminal state (done|abandoned) when receiving
      the new label
  (c) No other `status:*` label remains on the Issue/PR after the
      transition (intra-artifact mutex applied)
  (d) Label of the correct axis for the Issue/PR type:
      - Feature/fix/chore/refactor Issues/PRs → Axis A only
      - Release Issue (label `release ↗️`) → Axis B only
  (e) Issue body updated by the latest kata-flush-plan-to-issue
      (Axis A) or Tracks populated with PRs merged since the last
      tag (Axis B, transition `— → to release`)
  (f) Issue is not an Epic (Epic does not receive status:*)

This rule applies to EVERY Issue and EVERY PR in the Issue-Driven flow,
regardless of:
  - perceived size ("it is just a chore")
  - urgency ("production fire")
  - who requested ("the CEO asked")
  - team confidence ("we already tested a lot")

Declared exception: Epic and Issues generated by Dependabot/scanners
follow their own flow. Every other type of Issue/PR in the
Issue-Driven flow respects the mutex and the axis separation.
</HARD-GATE>
```

## Examples

### Correct — Axis A (feature Issue + PR)

```bash
# Eunomia creates Issue #96 with canonical body
gh issue edit 96 --add-label "status: todo"

# Athena enters Phase 4
gh issue edit 96 --remove-label "status: todo" --add-label "status: development"

# Athena opens PR #97; applies label simultaneously
gh pr create ... && gh pr edit 97 --add-label "status: to review"
gh issue edit 96 --remove-label "status: development" --add-label "status: to review"

# Argos starts review
gh pr edit 97 --remove-label "status: to review" --add-label "status: review"
gh issue edit 96 --remove-label "status: to review" --add-label "status: review"

# Argos finishes without approving; human pinged at 3×15min
gh pr edit 97 --remove-label "status: review" --add-label "status: to review"
gh issue edit 96 --remove-label "status: review" --add-label "status: to review"

# Human approves; merge closes Issue
gh pr edit 97 --remove-label "status: to review" --add-label "status: done"
gh issue edit 96 --remove-label "status: to review" --add-label "status: done"
```

### Correct — Axis B (release Issue)

```bash
# Janus opens release Issue #100; populates Tracks with PRs merged
# since the last tag
gh issue create --title "release: 0.4.0" \
  --body "Tracks: #93, #96, #98, #99" \
  --label "release ↗️,status: to release"

# Janus starts kata-release-prepare
gh issue edit 100 --remove-label "status: to release" --add-label "status: release"

# Janus completes: tag pushed, validate-tag.yml passes, Release created
gh issue edit 100 --remove-label "status: release" --add-label "status: done"
```

### Incorrect

```bash
# ❌ Two simultaneous status:* labels (violates intra-artifact mutex)
gh issue edit 96 --add-label "status: to review" --add-label "status: review"

# ❌ Cross-cycle labeling: status: to release on a feature Issue
gh issue edit 96 --add-label "status: to release"
# Issue #96 is a feature (Axis A); status: to release belongs to Axis B

# ❌ Reverse cross-cycle labeling: status: development on a release Issue
gh issue edit 100 --add-label "status: development"
# Issue #100 is a release Issue (Axis B); status: development belongs to Axis A

# ❌ Applying status:* on an Epic
gh issue edit 88 --add-label "status: development"
# Issue #88 has Issue Type Epic — forbidden per Rule 7
```

## Automated Validation

- **Tool:**
  - Script `scripts/bootstrap_labels.sh` creates the labels idempotently in any repository.
  - PR review (human or Argos) verifies:
    - Alignment between the Issue label and the PR label (same axis, same state).
    - Issue body reflects the latest flush (`kata-flush-plan-to-issue` was executed on the transition).
    - Correct axis applied to the Issue type (feature ⇒ Axis A; release Issue ⇒ Axis B).
  - Periodic verification GitHub Action (future) detects:
    - Issues/PRs with 0 or ≥2 `status:*` labels
    - Cross-cycle labeling (Axis A label on a release Issue, or Axis B label on a feature Issue)
    - Issues with Issue Type Epic carrying `status:*`
- **Timing:** when opening/updating an Issue, when opening/updating a PR, on each owner transition, at Gate 2 (`kata-quality-gate`).
- **Metric:** 0 Issues/PRs with cross-cycle `status:*` label; 0 Epics with `status:*`; 100% of transitions registered by the declared owner; 100% of release Issues with `Tracks:` populated.
