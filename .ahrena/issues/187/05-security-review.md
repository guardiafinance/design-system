## GitHub Actions workflow analysis

### Permissions

The `build` job (where the new steps were added) does not declare `permissions:` — therefore inherits the repository default (typically `contents: read`). The added steps (`npm run typecheck` and `npm run typecheck:docs`) read repository files and execute the local `tsc` binary; they don't need write permissions. **No permission escalation.**

### Workflow injection (CWE-94)

The added steps run static commands without interpolation of user-controlled context (`${{ github.event.* }}`). They don't read or pass PR body, title, labels, or branch names to the shell. **No injection surface.**

### Secret exposure

The new steps don't consume secrets. `npm ci` already runs in the previous step; no `NPM_TOKEN` or similar is referenced. **No secret exposure.**

### Pinned actions

The PR doesn't add new third-party actions. The existing actions (`actions/checkout@v4`, `actions/setup-node@v4`) remain unchanged.

## tsconfig analysis

The alias realignment from `"@/*": ["./src/*"]` to `"@/*": ["../ui_kit/*"]` changes the **type** resolution scope. It doesn't change the **runtime** resolution (which was already correct via Astro/Vite). Implications:

- No effect on the production bundle (the bundle isn't generated by this PR).
- No effect on transitive runtime dependencies.
- No effect on PII/credential exposure.
- Effect on the **type-safety surface**: more files (`ui_kit/components/**`) become indirectly type-checked under `docs/tsconfig.json`. This is the desired effect.
